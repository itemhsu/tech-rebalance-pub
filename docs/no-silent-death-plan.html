🔊 杜絕安靜失敗：統一動作日誌設計

目錄 一、安靜失敗的血案 · 二、設計原則 · 三、ActionLog 架構 · 四、三層曝光 · 五、禁止規則 · 六、改造清單 · 七、測試

一、安靜失敗的血案（為何需要這個）

案例 1：urllib SSL 在 DMG 內失敗
repo_sync 用 urllib 抓 template → 打包 App 內 SSL 驗證失敗 → except: return None → render 靜默跳過 → test_email.yml 從沒推上去。 使用者卻看到「✅ 建立完成」對話框 → 以為成功 → 測試發信 404。
致命點：失敗無 log、結果對話框謊報成功。
案例 2：manifest environments 壓成 list
probe_broker 對 list 呼叫 .get → 例外 → 被上層 except Exception 吞成 「連線失敗」一行字，沒有 traceback、沒有哪一步、沒有哪個值。除錯只能猜。
案例 3：gh 回非零卻被忽略
多處 _gh([...]) 不檢查 returncode 就往下走，後續才在別處爆， 錯誤訊息與真正失敗點相距十萬八千里。
共同病根：① 失敗被吞（bare except / return None）　② 結果不反映真實（假成功對話框）　③ 內部步驟無足跡。 本設計三者一次根治。

二、設計原則

  1. 每個按鈕動作都記 start 與 end(ok/fail)。
  2. 每個內部步驟（gh 呼叫、抓檔、PUT、sync 每個 path）都記一筆，含狀態與細節。
  3. 失敗一律有聲：例外 → 記 traceback + 重新拋或顯示；return None 前必先記原因。
  4. 結果對話框反映真實：依實際步驟結果統計，有任何 skip/fail 就顯示 ⚠ 並列出，絕不謊報 ✅。
  5. 可追蹤：同一次按鈕的所有子步驟共用一個 action_id，在日誌中成群顯示。
  6. 可見 + 可存：寫入持久 JSONL，並即時顯示在「日誌」分頁。

二之二、log 密度標準：讀 log 就能除錯，不用截圖

反例（密度不足，等於沒寫）：

   ⚠ 1/4 步有問題
看不出：哪個檔？為什麼 warn？gh 回什麼？URL 是什麼？版本多少？→ 還是得問使用者截圖。

每筆 step 的「最低資訊量」契約

動作類型detail 必含
gh 呼叫完整 args（金鑰除外）、rc、非零時 stderr 前 200 字
HTTP / 抓檔method、URL、status/rc、失敗原因（SSL/404/timeout）
檔案 PUTrepo、path、bytes、是否帶 sha（新建/覆蓋）、rc
sync 每個 pathpolicy、action 結果、若 skip/fail 必附原因（連到是哪一步失敗）
SMTP / 寄信host、port、收件人、login 帳號（密碼除外）、例外類別
例外例外型別 + 訊息 + traceback 末 1500 字 + 當下關鍵變數值
每個 action 開頭env 快照：app 版本、OS、gh 登入名、repo slug、引擎 pin
三條密度硬規則：
  1. 每個 action 第一筆 step 必為 env 快照（_ActionScope 自動補），讓每次操作都自帶環境。
  2. 任何 fail/skip/warn 的 detail 不得為空——必含「是什麼、為什麼、相關值」。lint 檢查空 detail 的非 ok step。
  3. gh / http 一律用包好的 logged 版，回傳即記 rc + err，呼叫端不可裸呼。

二之四、呼叫外部前，先記「資源就緒狀態」（preflight）

血案：「測試發信」直接觸發 GitHub workflow，呼叫前從沒檢查它依賴的資源。 workflow 跑 9 秒後才回 missing EMAIL_SENDER。 GUI log 只有 test_email: fail。於是猜了四輪（編碼/空格/IP/auth）全錯—— 因為根因（EMAIL_SENDER secret 不存在）在「呼叫之前」就已注定，卻從沒被記下。
原則：任何外部呼叫前，先 log 它依賴的「每一個資源」的就緒狀態。 缺資源 → 直接記成失敗原因，並在呼叫前就擋下（不盲目觸發、不等遠端回一個模糊錯誤）。 
def _test_email(self):
    with LOG.action("測試發信", ctx=self.repo_slug) as a:
        ready = preflight(a, self.repo_slug, requires=[
            Secret("EMAIL_SENDER"),            # ← 這次缺的，preflight 立刻抓到
            Secret("EMAIL_PASSWORD"),
            WorkflowFile("test_email.yml"),
            GhAuth(),
        ])
        if not ready:
            return    # 缺資源 → 不觸發；log 已寫明缺什麼
        ...觸發 workflow...
preflight 產生的 log（就緒一目了然）： 
▶ 測試發信  ctx=item3hsu/tech-rebalance
   · preflight secret EMAIL_SENDER: MISSING   ← 根因，呼叫前就現形
   · preflight secret EMAIL_PASSWORD: present
   · preflight workflow test_email.yml: present
   · preflight gh auth: ok (item3hsu)
   ⚠ 前置檢查未過：缺 secret EMAIL_SENDER → 未觸發

資源類型（每種都有 readiness 檢查）

資源檢查方式記什麼
Secretgh secret list 是否含此名present / MISSING（值不取，不外洩）
WorkflowFilerepo 的 .github/workflows/ 是否含此檔present / missing（含查的是哪個分支）
RepoExistsgh api repos/{slug}存在 / 404 / 無權限
GhAuthgh auth status / login 名登入名 / 未登入
Config 值本機 config 該鍵是否有值有值 / 空
EngineVersionpub release 列得到嗎最新版 / 列不到
Network / 抓檔對端是否可達ok / SSL / 404 / timeout
兩條硬規則：
  1. 每個外部呼叫前必有一段 preflight，列出它依賴的所有資源就緒狀態（即使全 ok 也要記，證明檢查過）。
  2. preflight 任一資源 MISSING → 記成失敗原因並中止，不把問題丟給遠端用模糊錯誤回報。
本案若有 preflight：第一次按「測試發信」就會顯示「缺 EMAIL_SENDER secret」，0 秒定案。

二之五、跨系統邊界：抓回「對方的原始回應」

GUI → GitHub Actions → Gmail SMTP，每跨一層就丟失下一層的回應。 GitHub workflow 印的 FAIL: missing EMAIL_SENDER、Gmail 回的 535-5.7.8 ... 都從沒進 GUI log，只剩一句 fail。
原則：凡跨系統邊界的呼叫，失敗時把對方的原始回應抓回來記。
邊界抓回什麼機制
觸發 GitHub workflow該 run 的 stdout / 失敗步驟全文失敗時自動 gh run view {id} --log 撈進 GUI log
workflow → Gmail SMTPSMTPException 完整訊息（含 5xx 碼）workflow 內 print(repr(e))，不吞成「寄信失敗」

broker RESTHTTP status + body 前段probe_broker 記回應，不只「連線失敗」

三、ActionLog 架構

3.1 核心：admin_gui/services/action_log.py

@dataclass
class Step:
    name: str; status: str; detail: str = ""    # status: ok | fail | skip | warn

class ActionLog:
    """單例：寫 JSONL + 發 Qt signal 給日誌分頁即時顯示。"""
    def __init__(self, path=AUDIT_JSONL): ...
    def emit(self, record: dict): ...            # 寫檔 + signal

    @contextmanager
    def action(self, name: str, ctx: str = ""):
        aid = uuid4().hex[:8]
        self.emit({"ts":now(), "aid":aid, "kind":"start", "action":name, "ctx":ctx})
        a = _ActionScope(self, aid, name)
        try:
            yield a
        except Exception as e:                    # 例外＝大聲記，不吞
            self.emit({"ts":now(),"aid":aid,"kind":"fail","action":name,
                       "error":repr(e), "trace":traceback.format_exc()})
            a._failed = True
            raise                                 # 不私吞，往上拋
        finally:
            self.emit({"ts":now(),"aid":aid,"kind":"end","action":name,
                       "ok": not a.has_problem(), "summary": a.summary()})

class _ActionScope:
    def step(self, name, status="ok", detail=""):  # 每個內部步驟
        self.steps.append(Step(name,status,detail))
        LOG.emit({"ts":now(),"aid":self.aid,"kind":"step","name":name,
                  "status":status,"detail":detail})
    def has_problem(self): return any(s.status in ("fail","warn","skip") for s in self.steps)
    def problems(self):   return [s for s in self.steps if s.status in ("fail","warn","skip")]

3.3 gh 與抓檔包一層，回傳即記

def gh_logged(args, a, inp=None):
    code, out, err = _gh(args, inp)
    if code != 0:
        a.step("gh "+args[0:2], "warn", f"rc={code} {err[:120]}")   # 非零＝有聲
    return code, out, err

四、四層曝光（看得到 + 存得住 + 寄得出）

層機制用途
① 即時對話框 / 狀態列action 結束依 problems() 顯示 ✅ 或 ⚠＋清單當下不被謊報成功
② 日誌分頁ActionLog signal → 表格即時 append（依 aid 分群、可展開步驟）追蹤一次操作的每一步
③ 持久 JSONL~/.tradingadmin/action_log.jsonl事後除錯
④ 📧 發送 log 按鈕日誌分頁一鍵把 log 寄給 itemhsu使用者回報問題、開發者遠端診斷

4.1 日誌分頁的「📧 發送 log」按鈕

用途：使用者遇到問題，按一下就把完整動作日誌寄給 itemhsu@gmail.com， 開發者不必猜、不必要求使用者複製貼上。

寄送內容

  • 最近 N 筆（預設 500）action_log.jsonl，已遮罩任何疑似金鑰/token（***）
  • 環境摘要：App 版本、OS、gh 登入名、repo slug、引擎 pin 版本
  • 主旨：[TradingAdmin log] {gh_login} {本地時間} {問題摘要可選}

寄送機制（與「測試發信」一致，零新依賴）

def _send_log_email(self):
    with LOG.action("發送 log email") as a:
        body = mask_secrets(read_log_tail(500)) + "\n\n" + env_summary()
        a.step("組裝 log", "ok", f"{len(body)} bytes")
        # 用使用者自己的 EMAIL_SENDER / EMAIL_PASSWORD（SMTP），收件人固定 itemhsu
        ok, msg = probes.send_log_to_dev(sender=EMAIL_SENDER, pw=EMAIL_PASSWORD,
                                         to="itemhsu@gmail.com", body=body)
        a.step("SMTP 寄送", "ok" if ok else "fail", msg)
    self._result_dialog(a)        # 成功/失敗都依實際結果，不假成功

沒有 EMAIL_PASSWORD 時的後援

若使用者尚未設 EMAIL_PASSWORD（無法 SMTP），按鈕改為：
  • 把 log 存成桌面檔 tradingadmin-log-YYYYMMDD.txt，並開啟 Finder 標示，請使用者手動寄給 itemhsu；或
  • 開 mailto:itemhsu@gmail.com?subject=...&body=...（短 log）由系統郵件 App 帶出。
安全：寄送前一律 mask_secrets()（比對 GitHub PAT / 長度≥20 的疑似金鑰 / *_SECRET= 樣式 → ***）； 收件人寫死 itemhsu，不接受 UI 輸入（避免被導向他處）。

4.2 日誌分頁的「🗑 清除 log」按鈕

用途：使用者重現問題前先清掉舊 log，寄來的 log 才乾淨、只含這次的足跡， 不會混入幾天前的雜訊讓除錯失焦。

建議流程（UI 引導）

重現問題標準步驟（顯示在分頁說明）：
  1. 按「🗑 清除 log」      → 清空
  2. 重做出問題的操作        → 只留這次
  3. 按「📧 發送 log」       → 寄乾淨的 log

行為

def _clear_log(self):
    with LOG.action("清除 log") as a:               # 清除本身也記一筆（諷刺但必要）
        n = LOG.clear()                              # truncate action_log.jsonl
        a.step("truncate action_log.jsonl", "ok", f"清除 {n} 行")
    # 清除後第一筆就是這個「清除」action → 標記新一輪的起點
    self.refresh()
  • 只清 action_log.jsonl（本機動作日誌）；不碰排程執行歷史（那在 GitHub，且非本機）。
  • 清除前彈確認框（不可逆）。
  • 清除動作本身寫一筆 log（含清掉幾行）→ 連「清空」都不靜默，且成為新一輪的時間起點。
  • 可選的自動清理：log 超過上限（如 5000 行 / 2 MB）時自動 rotate 保留尾段，避免無限長。
結果對話框新規則（取代「✅ 建立完成」）： 
problems = a.problems()
if not problems:
    info("✅ 完成", "全部步驟成功")
else:
    warn("⚠ 完成但有問題", "\n".join(f"- {p.name}: {p.status} {p.detail}" for p in problems))
test_email.yml 抓不到範本 → 對話框直接列「- repo_b/test_email.yml: skip 取不到範本」， 使用者與我們都立刻知道，不會再被假成功騙。

五、禁止規則（可被 CI lint）

規則說明例外
R1 禁 bare exceptexcept: / except Exception: pass 一律禁須 a.step(..., "fail", repr(e)) 或 re-raise
R2 return None 前必記任何「失敗回 None」前先 a.step(name,"fail/skip",why)—
R3 gh 非零必記所有 _gh 呼叫經 gh_logged 或自行檢查 rc—
R4 對話框不得假成功顯示成功前必查 problems()—
R5 每個 clicked.connect 的 handler 都包 action()introspection 測試掃描所有按鈕純導覽（切分頁）可豁免
R6 非 ok step 的 detail 不得為空fail/skip/warn 必含「是什麼+為什麼+值」—
R7 每個 action 自帶 env 快照第一筆 step 為 app/os/gh/repo/pin—
R8 外部呼叫前必有 preflight列出所依賴資源就緒狀態；缺則中止純本機計算可免
R9 跨邊界失敗抓回對方原始回應workflow stdout / SMTP 5xx / HTTP body—
Lint 實作：scripts/check_no_silent.py（pub engine + GUI 各一份 CI）： AST 掃 except 子句，若 body 只有 pass 或只有 return None 且同 scope 無 log 呼叫 → 報錯。
